# CARLA OpenDRIVE Compatibility Investigation

Investigation into why CARLA's path planner cannot find all connections in roundabout
XODR files exported by ORBIT, even though the connections are visible in the file.

## Root Cause: Road ID / Junction ID Collision

### The problem

ORBIT uses **separate ID counters** for roads and junctions, both starting at 1:

```python
# orbit/models/project.py
_next_road_id: int = field(default=1, repr=False)
_next_junction_id: int = field(default=1, repr=False)
```

This guarantees that exported XODR files will have road IDs that overlap with junction
IDs. For example, in `SaroRound_2laneU.xodr`:

| Road IDs | 1, 3, 6, 15, 21–38, 45–49 |
|----------|---------------------------|
| Junction IDs | 1, 2, 3, 4, 5, 6 |
| **Overlap** | **1, 3, 6** |

### Why CARLA breaks

CARLA's OpenDRIVE road parser (`LibCarla/source/carla/opendrive/parser/RoadParser.cpp`)
stores `predecessor` and `successor` as bare unsigned integers, **discarding the
`elementType` attribute**:

```cpp
road.predecessor = link.child("predecessor").attribute("elementId").as_uint();
road.successor = link.child("successor").attribute("elementId").as_uint();
```

When resolving links at runtime, `MapBuilder::GetLaneNext()` determines whether a
link target is a road or junction using:

```cpp
bool next_is_junction = !_map_data.ContainsRoad(next_road);
```

If a road exists with the same numeric ID as a junction, CARLA **always** treats the
link as road-to-road, completely bypassing the junction and all its connections.

### Impact

In `SaroRound_2laneU.xodr`, 9 out of 18 road-to-junction links (50%) are
misinterpreted. Three of six roundabout junctions (1, 3, 6) are completely unreachable:

```
Road 26 (Ring 6) has successor = junction 1
Road 1 (Valldavägen) exists with id = 1
→ CARLA: ContainsRoad(1) = true → treats as road-to-road → junction 1 never entered
```

Affected links in detail:

| Road | Link | Intended target | CARLA misinterprets as |
|------|------|----------------|----------------------|
| 1 (Valldavägen) | predecessor=1 | Junction 1 | Road 1 (self!) |
| 3 (Säröleden) | predecessor=1 | Junction 1 | Road 1 |
| 21 (Ring 1) | predecessor=1 | Junction 1 | Road 1 |
| 22 (Ring 2) | successor=3 | Junction 3 | Road 3 |
| 23 (Ring 3) | predecessor=3 | Junction 3 | Road 3 |
| 25 (Ring 5) | successor=6 | Junction 6 | Road 6 |
| 26 (Ring 6) | predecessor=6 | Junction 6 | Road 6 |
| 26 (Ring 6) | successor=1 | Junction 1 | Road 1 |
| 45 (Säröleden) | successor=6 | Junction 6 | Road 6 |

Junctions 2, 4, 5 work correctly because no road shares their IDs.

### Reference file comparison

`saro_fixed_bidirectional.xodr` (generated by SUMO/netconvert) uses road IDs 190–248
and junction IDs 1–17 — **zero overlap**. This file works in CARLA.

### Fix applied

Road IDs are now remapped during export to avoid junction ID collisions. Internal
project IDs are unchanged — only the exported XML values are affected.

## Secondary Findings

### 1. Geometry gaps at connecting road endpoints

Both ORBIT and the reference file have small geometry gaps (1–6 meters) between
connecting road endpoints and the roads they link to. The reference file has gaps
of 1–4m; ORBIT files have gaps of 1–6m.

**Implication**: Not the cause of the CARLA routing failure (the reference file has
the same gaps and works). However, larger gaps could cause visual artifacts or waypoint
placement issues in CARLA. The endpoint snapping in `_snap_cr_endpoints_to_roads()`
could be improved.

### 2. No junctionGroup elements for roundabouts

ORBIT does not export `<junctionGroup>` elements even when junctions are tagged as
roundabout junctions (`is_roundabout=True`). The reference file also lacks them.

**Implication**: Not required by CARLA for basic routing. However, junction groups
provide metadata that some tools use to treat the roundabout as a single entity (e.g.,
for high-level routing decisions or traffic management). Could be added as a future
improvement.

### 3. OpenDRIVE version and XML namespace

ORBIT exports OpenDRIVE 1.8 with `xmlns="http://code.asam.net/simulation/standard/opendrive_schema"`
on the root element. The reference file uses OpenDRIVE 1.4 with no namespace.

**Implication**: Not a problem for CARLA — pugixml (CARLA's XML parser) ignores default
namespaces and matches element names by local name only. However, some older OpenDRIVE
tools or validators may not handle the namespace correctly.

### 4. Route coverage is complete

When ignoring the ID collision issue, ORBIT's junction topology is correct: every
incoming road can reach every outgoing road through the junction connections. The
`contactPoint`, `incomingRoad`, `connectingRoad`, and `laneLink` attributes are all
semantically correct.

### 5. CARLA does not parse contactPoint from junction connections

CARLA's `JunctionParser.cpp` does not read the `contactPoint` attribute from
`<connection>` elements. It only reads `id`, `incomingRoad`, and `connectingRoad`.
The contact point is inferred from the connecting road's geometry and link structure.
